# AI Agent Ender

이 문서는 공통 세션 종료 규약이다.
사용자가 아래처럼 말하면 현재 역할 세션은 이 문서를 종료 프로토콜로 즉시 적용해야 한다.

```text
세션 종료할꺼야
ender.md 읽어 세션 종료할꺼야
```

이 문서는 새로운 역할을 정하는 문서가 아니다.
최상위 세션의 역할은 `starter.md` 에 의해, 서브 에이전트의 역할은 각 `docs/agent/<role>/README.md` 진입 문서에 의해 이미 고정되어 있다고 가정한다.
종료 시점에는 **현재 역할 그대로** 자기 역할 종료 절차만 수행한다.

## 1. 이 문서의 목적

이 문서는 종료 직전에 아래를 강제한다.

- 자기 역할 문서만 마지막으로 정리한다.
- 종료 유형을 `normal-cycle-close` 와 `interrupted-handoff-close` 로 먼저 분류한다.
- 불필요한 기록은 삭제하거나 압축한다.
- 다음 작업에 필요한 핵심 정보만 남긴다.
- 사용자에게 줄 종료 요약은 짧고 선명하게 남긴다.
- 역할 침범 없이 세션을 닫는다.

## 2. 공통 종료 규칙

모든 역할은 종료 직전에 아래를 반드시 지킨다.

1. 현재 역할을 유지한다.
2. 종료 정리 외의 새 작업을 벌리지 않는다.
3. 자기 역할 허용 범위 밖의 문서는 수정하지 않는다.
4. 종료 유형에 맞는 자기 역할 문서만 최신 상태로 맞춘다.
5. 끝난 항목, 중복 기록, 의미 없는 로그는 삭제하거나 압축한다.
6. 다음 판단이나 다음 세션에 필요한 핵심 정보만 남긴다.
7. 불확실한 내용은 확정된 사실처럼 남기지 않는다.
8. 종료 응답에는 실제로 반영된 내용만 말한다.

종료 시점의 문서 정리는 "많이 남기는 것"이 목적이 아니다.
핵심만 남기고, 필요 없는 찌꺼기를 치우는 것이 목적이다.

## 3. 종료 트리거 해석 규칙

종료 트리거를 받으면 아래처럼 이해한다.

- `"세션 종료할꺼야"`: 현재 역할 기준으로 `ender.md` 절차를 수행하라는 뜻
- `"ender.md 읽어 세션 종료할꺼야"`: `ender.md` 절차를 명시적으로 수행하라는 뜻

이 경우 세션은 사용자에게 다시 역할을 물을 필요가 없다.
이미 시작 시점에 고정된 현재 역할만 기준으로 종료한다.

단, 역할 없이 `starter.md` 의 기본 부트스트랩 오케스트레이션 모드로 시작한 세션은 현재 역할을 `orchestra` 로 본다.
그 외에 현재 세션의 역할이 애초에 정해진 적이 없다면 종료 절차를 진행하지 말고 역할부터 다시 확인해야 한다.

## 4. 종료 유형 분류

종료 트리거를 받으면 역할별 종료 절차를 실행하기 전에 먼저 종료 유형을 분류한다.
이 분류는 문서화 책임을 결정한다.
여기서 사용자가 말하는 `human-docs` 는 현재 문서 체계의 `user-docs` 역할과 같은 의미로 본다.

### 사용자 재시작/종료 지시 시점 기준

사용자가 세션 재시작 또는 세션 종료를 지시하면, 그 시점의 작업 상태를 기준으로 아래처럼 처리한다.

1. 그 시점 기준으로 최종 `planning` 보고와 사용자 반응까지 도달했고 정상 문서화 단계로 닫을 수 있으면 `normal-cycle-close` 로 처리한다.
2. `normal-cycle-close` 에서는 `user-docs` 와 `ai-docs` 가 각각 자기 역할에 맞게 문서화 업데이트를 수행하고, 두 문서 갱신까지 끝나야 1회 개발 사이클 완료로 본다.
3. 그 시점이 정상적인 1회 작업 사이클 완료 전이고, 아직 하던 작업 도중이면 `interrupted-handoff-close` 로 처리한다.
4. `interrupted-handoff-close` 에서는 `ai-docs` 와 `user-docs` 를 사용해 전체 상황을 대신 문서화하지 않는다.
5. `interrupted-handoff-close` 에서는 그 시점에 활성화되어 있는 모든 역할군 에이전트가 직접 자기 작업을 문서화한다. 이 대상에는 최상위 `orchestra` 도 포함한다.
6. 각 활성 역할은 자기 역할 폴더의 `docs/agent/<role>/handoff/latest.md` 에만 인계를 남긴다.
7. 이 인계는 다음 세션에서 바로 이어서 작업할 수 있을 정도로 구체적이어야 한다.
8. 구체적 인계에는 현재 작업, 현재 루프 단계, 완료한 일, 진행 중이던 일, 다음 행동, 수정/확인한 파일, 아직 검수되지 않은 변경, 막힌 질문, 자기 `Context Report` 를 포함한다.
9. 어떤 역할이 이미 닫혔거나 응답하지 못해 자기 인계를 남기지 못하면, `orchestra` 는 그 사실만 자기 인계에 기록하고 해당 역할의 세부 작업을 추측으로 대신 작성하지 않는다.

### A. `normal-cycle-close`

아래 조건이면 정상 사이클 종료로 본다.

- 최종 `planning` 보고와 사용자 반응까지 도달했고 정상 문서화 단계로 닫을 수 있다.
- `user-docs` 와 `ai-docs` 가 정상적으로 각자의 문서 갱신을 수행할 수 있다.
- 현재 작업이 대체로 정리되어 있고, 다음 세션은 문서화된 현재 상태에서 새 작업 또는 다음 사이클로 시작하면 된다.

이 경우 문서화 책임은 기존 개발 사이클 규칙을 따른다.

1. `user-docs` 는 정상 작업 범위인 `README.md` 와 `docs/human/` 의 사용자용 문서를 갱신한다. 기본 작업 공간은 `docs/human/user-docs/` 이다.
2. `ai-docs` 는 AI 에이전트용 개발 문서와 역할 문서를 갱신한다.
3. 각 역할은 자기 정상 관리 문서만 정리한다.
4. `codegen` 은 정상 종료에서는 문서를 작성하거나 갱신하지 않는다.

### B. `interrupted-handoff-close`

아래 조건 중 하나라도 해당하면 중간 중단 인계 종료로 본다.

- 최상위 `orchestra` 또는 주요 서브 에이전트의 컨텍스트가 한계에 가까워 세션 재시작이 필요하다.
- 개발 루프가 아직 진행 중이고, 마지막 `planning` 보고와 `user-docs`/`ai-docs` 정상 문서 갱신 단계에 도달하지 못했다.
- 사용자가 작업 도중 세션을 닫거나 재시작하겠다고 말했다.
- 서브 에이전트가 닫히면 각자 하던 작업의 세부 맥락이 사라질 위험이 있다.

이 경우 `ai-docs` 와 `user-docs` 가 모든 역할의 작업을 대신 문서화하지 않는다.
각 역할이 자기 작업을 가장 잘 알고 있으므로, 각 역할이 자기 역할 폴더 아래의 인계 문서만 직접 남긴다.

중간 중단 인계 규칙:

1. 종료 정리 외의 새 구현, 새 검수, 새 테스트, 새 계획을 시작하지 않는다.
2. 새 서브 에이전트를 만들지 않는다.
3. 현재 활성 역할은 자기 역할 폴더의 `handoff/latest.md` 를 작성하거나 갱신한다.
4. 파일 경로는 항상 `docs/agent/<role>/handoff/latest.md` 이다.
5. `orchestra` 는 자기 인계를 `docs/agent/orchestra/handoff/latest.md` 에 남긴다.
6. `orchestra` 는 가능하면 아직 살아 있는 각 서브 에이전트에게 자기 역할의 `handoff/latest.md` 를 작성하고 종료하라고 지시한다.
7. 이미 닫혔거나 응답하지 않는 서브 에이전트가 있으면 `orchestra` 는 그 사실만 자기 인계 문서에 남기고, 해당 역할의 작업을 대신 꾸며 쓰지 않는다.
8. `codegen` 은 이 경우에만 예외적으로 `docs/agent/codegen/handoff/latest.md` 를 작성하거나 갱신할 수 있다. 다른 문서는 여전히 수정 금지다.
9. `user-docs` 는 중간 중단 인계 종료에서는 `README.md` 나 `docs/human/` 을 갱신하지 않고, 자기 인계만 `docs/agent/user-docs/handoff/latest.md` 에 남긴다.
10. `ai-docs` 는 중간 중단 인계 종료에서는 전체 역할 문서를 대신 정리하지 않고, 자기 인계만 `docs/agent/ai-docs/handoff/latest.md` 에 남긴다.
11. 이 종료 유형에서는 1회 개발 사이클이 완료되었다고 쓰지 않는다.

모든 역할의 `handoff/latest.md` 는 아래 형식을 따른다.

```markdown
# <role> Interrupted Handoff

- role:
- handoff_type: interrupted
- last_updated:
- trigger:
- current_task:
- current_loop_step:
- completed:
- in_progress:
- next_steps:
- touched_files:
- unverified_changes:
- blocking_questions:
- context_report:
```

`completed` 에는 실제로 끝난 것만 적고, `in_progress` 에는 다음 세션이 바로 이어야 할 상태를 적는다.
`unverified_changes` 에는 아직 검수나 테스트가 끝나지 않은 변경을 반드시 분리한다.

역할별 종료 절차의 관리 문서는 종료 유형에 따라 다르게 적용한다.
`normal-cycle-close` 에서는 각 역할의 정상 관리 문서를 정리하고, `interrupted-handoff-close` 에서는 정상 관리 문서 갱신을 멈춘 뒤 자기 `handoff/latest.md` 만 작성한다.
특히 `user-docs` 와 `ai-docs` 는 중간 중단 인계 종료에서 정상 문서 갱신 역할을 수행하지 않는다.

## 5. 역할별 종료 절차

### `analysis`

이 역할은 종료 직전에 테스트 문제점의 근본 원인 분석 결과만 남긴다.

관리 문서:

- `docs/agent/analysis/README.md`
- 필요 시 `docs/agent/analysis/` 안의 짧은 분석 메모
- 중간 중단 인계 시 `docs/agent/analysis/handoff/latest.md`

반드시 할 일:

1. 실제로 확인한 테스트 문제점 리스트와 관련 파일 범위를 정리한다.
2. 근본 원인 후보, 수정 후보 파일, 다음 테스트 확인 지점만 남긴다.
3. 추측과 확인된 사실을 구분한다.
4. 구현, 검수 판정, 테스트 통과 단정을 남기지 않는다.
5. 최종 응답에는 orchestra 가 이어받을 분석 요약만 짧게 말한다.

하면 안 되는 일:

- 코드 수정
- 다른 역할 문서 수정
- 계획 확정
- 테스트 통과 단정

### `codegen`

이 역할은 문서화를 하지 않는다.
정상 종료 시에도 문서 갱신을 하지 않는다.
단, `interrupted-handoff-close` 에서는 다음 세션이 코드 작업을 이어받을 수 있도록 `docs/agent/codegen/handoff/latest.md` 만 작성하거나 갱신할 수 있다.

반드시 할 일:

1. 현재 턴에서 하던 구현이 자기 내부 기준으로 끊겼는지 확인한다.
2. 자신이 이번 세션에서 건드린 코드가 명백한 임시 찌꺼기 상태로 남지 않았는지 확인한다.
3. 안전하게 정리 가능한 자기 임시 흔적만 정리한다.
4. 새 기능 기획, 리뷰 판단, 사용자 문서 작성을 시작하지 않는다.
5. 종료 응답에는 이번 세션에서 실제로 바뀐 코드와 남은 코드 작업만 짧게 말한다.

하면 안 되는 일:

- 정상 종료에서 `docs/` 아래 어떤 문서도 수정
- 중간 중단 인계 종료에서 `docs/agent/codegen/handoff/latest.md` 밖의 어떤 문서도 수정
- orchestra/analysis/review/test/planning/ai-docs/ephemeral/user-docs 역할 문서 업데이트
- 종료 직전에 새 범위 구현 시작

### `review`

이 역할은 종료 직전에 자기 검수 기록을 정리해야 한다.
그리고 검수 결과는 반드시 하네스 기준 검증 결과와 함께 남겨야 한다.

관리 문서:

- `docs/agent/review/scope.md`
- `docs/agent/review/checklist.md`
- `docs/agent/review/findings.md`
- `docs/agent/review/fix-log.md`
- 중간 중단 인계 시 `docs/agent/review/handoff/latest.md`

반드시 할 일:

1. 이번 세션에서 실제로 본 리뷰 범위를 `scope.md` 에 최신 상태로 맞춘다.
2. 이번 세션에서 실제로 점검한 항목만 `checklist.md` 에 남긴다.
3. 실제로 발견한 문제만 `findings.md` 에 남긴다.
4. 실제로 수정한 내용, 하네스 검증 내용, 남은 이슈를 `fix-log.md` 에 정리한다.
5. 종료된 항목, 이미 해소된 발견 사항, 반복 로그는 삭제하거나 압축한다.
6. 검수나 수정이 있었다면 하네스 검증 결과가 문서에 반영되어 있는지 확인한다.
7. 최종 응답에는 핵심 발견 사항 또는 "이번 세션 기준 추가 findings 없음"을 분명하게 말한다.

반드시 지킬 검증 규칙:

- 수정 검증과 회귀 확인은 반드시 `crates/xavi-harness/` 를 통해 수행한다.
- 하네스 검증 없이 "확인됨", "안전함", "통과" 같은 표현을 쓰지 않는다.
- 하네스에서 확인한 사실만 남긴다.

### `test`

이 역할은 종료 직전에 실행한 검증 명령과 문제점 리스트만 남긴다.

관리 문서:

- `docs/agent/test/README.md`
- 필요 시 `docs/agent/test/` 안의 짧은 테스트 결과 메모
- 중간 중단 인계 시 `docs/agent/test/handoff/latest.md`

반드시 할 일:

1. 실행한 명령과 결과를 정리한다.
2. 실패가 있으면 재현 명령, 핵심 오류, 문제점 리스트를 남긴다.
3. 실패의 근본 원인 판단은 남기지 않고 `analysis` 가 이어받을 수 있는 입력만 남긴다.
4. 재테스트를 했다면 재테스트 결과를 분리해 남긴다.
5. 최종 응답에는 통과/실패와 남은 조치만 짧게 말한다.

하면 안 되는 일:

- 제품 코드 직접 수정
- 검수 판정 대체
- 기획 결정 변경
- 사용자 문서 작성

### `orchestra`

이 역할은 종료 직전에 사용자 대화, 서브 에이전트 상태, 남은 조율 사항만 남겨야 한다.

관리 문서:

- `docs/agent/orchestra/README.md`
- 필요 시 `docs/agent/orchestra/` 안의 짧은 운영 메모
- 중간 중단 인계 시 `docs/agent/orchestra/handoff/latest.md`

반드시 할 일:

1. 이번 세션에서 사용자와 합의한 내용과 아직 미결인 내용을 정리한다.
2. 실행한 서브 에이전트, 반환 요약, `Context Report`, 종료/재생성 필요 여부를 압축한다.
3. 아직 반환되지 않은 서브 에이전트 작업이 있으면 진행 중인지, 중단해야 하는지, 다음 세션이 이어야 하는지 명확히 남긴다.
4. 중간 중단 인계 종료라면 각 활성 서브 에이전트가 자기 `handoff/latest.md` 를 남기게 하고, 불가능한 경우 그 사실을 자기 인계에 남긴다.
5. 다음 `orchestra` 세션이 바로 이어받을 핵심 입력만 남긴다.
6. 종료 시점에는 새 서브 에이전트를 실행하지 않는다.
7. 최종 응답에는 확정 사항, 남은 작업, 필요한 사용자 입력만 짧게 말한다.

하면 안 되는 일:

- 종료 시점에 직접 구현 시작
- planning/codegen/review/test/analysis/ai-docs/user-docs 결과를 자기 판단처럼 꾸며 쓰기
- 단, 서브 에이전트가 반환한 요약은 사실 범위 안에서 압축해 전달할 수 있음

### `planning`

이 역할은 종료 직전에 계획 초안, 범위, 결정 후보, 미결만 남겨야 한다.

관리 문서:

- `docs/agent/planning/brief.md`
- `docs/agent/planning/dialogue-log.md`
- `docs/agent/planning/decision-log.md`
- 중간 중단 인계 시 `docs/agent/planning/handoff/latest.md`

반드시 할 일:

1. 이번 세션에서 정리된 목표, 범위, 제외 범위를 `brief.md` 에 반영한다.
2. `orchestra` 가 전달한 사용자 의도와 계획 관련 대화 중 실제로 의미 있는 내용만 `dialogue-log.md` 에 요약한다.
3. 확정된 결정, 보류된 결정, 열린 질문을 `decision-log.md` 에 분리해서 남긴다.
4. 단순 잡담, 반복된 표현, 결정에 영향 없는 대화 찌꺼기는 지우거나 압축한다.
5. 다음 `planning` 세션이나 `orchestra` 가 받아야 할 핵심 입력만 남긴다.
6. 계획 초안, 변경된 계획, 미결 질문 후보만 압축해 정리한다.
7. 최종 응답에는 계획 요약과 아직 필요한 입력만 짧게 말한다.

하면 안 되는 일:

- 종료 시점에 구현 방향을 임의로 확정
- review 판단이나 codegen 결과를 자기 판단처럼 대신 작성
- 사용자와 직접 장기 대화 주도
- 서브 에이전트 생성/종료/재생성 관리

### `ephemeral`

이 역할은 자기 세션 폴더 안에서만 종료 정리를 해야 한다.
임시 세션답게 결과만 남기고 찌꺼기는 치운다.

관리 문서:

- `docs/agent/ephemeral/RESULT_TEMPLATE.md`
- `docs/agent/ephemeral/sessions/<session-id>/...`
- 중간 중단 인계 시 `docs/agent/ephemeral/handoff/latest.md`

반드시 할 일:

1. 자기 세션 폴더 문서를 `RESULT_TEMPLATE.md` 구조에 맞게 정리한다.
2. 왜 실행되었는지, 무엇을 했는지, 무엇이 남았는지, 다음 소유자가 누구인지 남긴다.
3. 임시 탐색 메모, 실패한 시도 로그, 중복 조사 메모는 압축하거나 삭제한다.
4. 실제 결과와 아직 못 한 일을 명확히 구분한다.
5. 최종 응답에는 이번 임시 세션의 결과와 다음에 이어야 할 핵심만 남긴다.

하면 안 되는 일:

- 다른 ephemeral 세션 폴더 수정
- 장기 역할 문서 대체
- 자기 세션 범위를 넘는 작업 문서 정리

### `user-docs`

이 역할은 사람용 문서를 마지막으로 정리하고, 실제 존재하는 내용만 남겨야 한다.
핵심은 많이 쓰는 것이 아니라 읽기 쉽게 정리하는 것이다.

진입 확인 문서:

- `docs/agent/user-docs/README.md`

관리 문서:

- `README.md`
- `docs/human/README.md`
- `docs/human/user-docs/live-log.md`
- `docs/human/user-docs/file-map.md`
- `docs/human/user-docs/feature-map.md`
- 필요 시 새로 만든 사용자 보고서 문서
- 중간 중단 인계 시에는 위 사용자 문서를 갱신하지 않고 `docs/agent/user-docs/handoff/latest.md`

반드시 할 일:

1. `docs/agent/user-docs/README.md` 를 먼저 읽고 사용자 문서 역할 경계를 재확인한다.
2. 이번 세션에서 실제로 확인한 코드와 문서 변경만 반영한다.
3. 공개 개요나 주제 설명이 바뀌었으면 `README.md` 와 `docs/human/README.md` 를 현재 코드 상태와 맞춘다.
4. `live-log.md` 에는 최근 변경 중 사용자에게 의미 있는 항목만 남긴다.
5. `file-map.md` 에는 실제 파일 역할과 책임만 남긴다.
6. `feature-map.md` 에는 실제 구현된 기능 흐름과 로직만 남긴다.
7. 중복 설명, 오래된 설명, 코드와 어긋난 설명은 정리하거나 삭제한다.
8. 없는 기능, 추정 동작, 확인되지 않은 미래 계획은 문서에서 제거한다.
9. 사람이 빨리 읽을 수 있게 개요, 파일 역할, 기능 흐름, 용어 풀이 순으로 정리한다.
9. 최종 응답에는 어떤 사용자 문서를 갱신했는지와 사용자 입장에서 중요한 변경만 말한다.

반드시 지킬 사실성 규칙:

- 코드와 문서에서 확인한 사실만 쓴다.
- 확인되지 않은 것은 확인되지 않았다고 쓴다.
- 누락을 줄이되, 사람을 지치게 할 정도로 장황하게 쓰지 않는다.
- 내부 AI 메모를 사용자 문서에 섞지 않는다.

### `ai-docs`

이 역할은 AI 에이전트용 개발 문서를 마지막으로 정리하고, 다음 에이전트가 이어받을 정보만 남겨야 한다.

관리 문서:

- `docs/agent/ai-docs/README.md`
- 필요 시 `starter.md`
- 필요 시 `ender.md`
- 필요 시 `inject_subject_once.md`
- 필요 시 `docs/agent/` 아래 역할 문서
- 중간 중단 인계 시에는 위 운영 문서를 대신 정리하지 않고 `docs/agent/ai-docs/handoff/latest.md`

반드시 할 일:

1. 1회 개발 사이클의 역할별 산출물, 문제점 리스트, 근본 원인 분석, 테스트 결과, `planning` 보고서, 사용자 반응을 AI 인계 정보로 압축한다.
2. AI 에이전트가 다음 사이클에서 지켜야 할 제약, 미결 사항, 반복 문제만 남긴다.
3. 사용자용 설명, 튜토리얼, 보고서 문체는 쓰지 않는다.
4. `docs/human/` 은 읽거나 수정하지 않는다.
5. 최종 응답에는 어떤 AI 문서를 갱신했는지만 말한다.

하면 안 되는 일:

- 사용자용 문서 작성
- 제품 코드 수정
- 검수 판정 대체
- 테스트 통과 단정

## 6. 종료 응답 원칙

종료 응답은 길게 쓰지 않는다.
각 역할은 자기 역할 관점에서 아래만 짧게 말하면 된다.

- 적용한 종료 유형
- 무엇을 정리했는가
- 어떤 문서를 갱신했는가
- 무엇이 남았는가

단, `codegen` 은 문서 대신 코드 상태만 말한다.
`interrupted-handoff-close` 에서만 `codegen` 은 자기 인계 문서 경로도 말할 수 있다.
`user-docs` 는 정상 사이클 종료에서는 사용자 문서 갱신 결과만 말하고, 중간 중단 인계 종료에서는 자기 인계 문서 경로만 말한다.
`ai-docs` 는 정상 사이클 종료에서는 AI 에이전트용 문서 갱신 결과만 말하고, 중간 중단 인계 종료에서는 자기 인계 문서 경로만 말한다.

## 7. 한 줄 요약

최상위 세션의 시작 규약이 `starter.md` 라면,
`ender.md` 는 세션 종료 직전에 각 역할이 **정상 사이클 종료인지 중간 중단 인계 종료인지 먼저 분류하고, 자기 문서와 자기 결과만 깔끔하게 정리하고 닫게 만드는 종료 규약**이다.
