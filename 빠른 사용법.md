# 빠른 사용법

이 저장소는 "AI에게 전체 프로젝트를 한 번에 맡기는 템플릿"이 아닙니다.

핵심은 개발을 `1회 작업 사이클` 단위로 끊고,
각 사이클마다 사람이 계획, 결과, 테스트, 보고서를 확인하는 것입니다.

## 핵심 아이디어

AI 작업은 길어질수록 추적이 어려워집니다.

- AI가 처음 요청을 어떻게 이해했는지 흐려집니다.
- 어떤 지시 때문에 코드가 바뀌었는지 찾기 어려워집니다.
- 검수와 테스트가 실제로 있었는지 확인하기 어려워집니다.
- 결과가 마음에 안 들어도 어디서부터 고쳐야 할지 애매해집니다.

그래서 이 부트스트랩은 일을 작게 자릅니다.

```text
큰 목표 하나
-> 작은 작업 사이클 여러 개
-> 각 사이클마다 계획, 구현, 검수, 테스트, 보고서 확인
```

## 1회 작업 사이클

`1회 작업 사이클`은 작은 개발 목표 하나를 끝까지 확인하는 단위입니다.

```text
사용자 요청
-> orchestra가 이해하고 작업 분해
-> planning이 계획 작성
-> 사용자가 계획 승인
-> codegen이 구현
-> review가 검수
-> test가 실행과 검증
-> user-docs / ai-docs가 문서 정리
-> cycle-report가 HTML 보고서 생성
-> 사용자가 보고서 확인
-> 다음 사이클 지시
```

중요한 기준은 이것입니다.

**코드가 만들어졌다고 끝이 아닙니다.**

HTML 보고서를 보고 아래를 확인해야 한 사이클이 끝납니다.

- AI가 내 요청을 제대로 이해했는가?
- 승인한 계획대로 구현됐는가?
- 어떤 역할이 어떤 지시를 받았는가?
- 어떤 코드가 바뀌었는가?
- 검수와 테스트가 통과했는가?
- 누락된 증거나 경고가 있는가?

## 어떻게 활용하나

### 1. 처음에는 프로젝트 주제만 말합니다

```text
starter.md 읽고 문서 체계 숙지하고, 프로젝트 구조를 간단히 파악해줘.
```

그 다음 만들 프로젝트를 설명합니다.

```text
이 부트스트랩으로 개인 문서 요약 앱을 시작하고 싶어.

사용자:
개인 작업자

처음 필요한 기능:
문서 업로드, 요약 생성, 결과 저장

starter.md 기준으로 필요한 절차대로 진행해줘.
```

### 2. 큰 목표를 바로 구현시키지 않습니다

안 좋은 요청:

```text
문서 요약 앱 전체를 다 만들어줘.
```

좋은 요청:

```text
이번 사이클에서는 문서 업로드와 저장 구조까지만 구현해줘.
계획 먼저 보고하고, 내가 승인하면 진행해줘.
```

좋은 사이클은 하나의 작은 목표만 다룹니다.

- 화면 하나
- API 하나
- 데이터 구조 하나
- 버그 하나
- 테스트 보강 하나
- 문서 정리 하나

### 3. 계획을 보고 승인합니다

AI는 바로 구현하지 않고 먼저 계획을 내야 합니다.

사용자는 계획을 보고 이렇게 판단합니다.

- 범위가 너무 큰가?
- 이번 사이클에서 확인 가능한가?
- 빠진 요구사항이 있는가?
- 구현 전에 바꿔야 할 방향이 있는가?

계획이 괜찮으면 승인합니다.

```text
이 계획대로 진행해줘.
```

계획이 크면 줄입니다.

```text
범위가 커.
이번 사이클에서는 업로드 API까지만 하고,
화면은 다음 사이클로 미뤄줘.
```

### 4. HTML 보고서를 보고 다음 지시를 냅니다

사이클이 끝나면 HTML 보고서를 확인합니다.

보고서는 다음 지시를 만들기 위한 기준입니다.

예시:

```text
보고서를 보니 업로드 API와 저장 구조는 괜찮아.
다음 사이클에서는 업로드한 파일 목록 화면만 만들어줘.
계획 먼저 보고하고, 내가 승인하면 진행해줘.
```

또는:

```text
보고서에서 테스트가 부족해 보여.
다음 사이클에서는 업로드 실패 케이스 테스트만 보강해줘.
```

## 역할을 이렇게 이해하면 됩니다

- `orchestra`: 사용자와 대화하고 작업을 역할별로 나눕니다.
- `planning`: 계획과 최종 완성도 보고를 만듭니다.
- `codegen`: 코드를 구현합니다.
- `review`: 구현 결과를 검수합니다.
- `test`: 실행과 테스트를 확인합니다.
- `user-docs`: 사용자가 볼 문서를 정리합니다.
- `ai-docs`: 다음 AI 작업에 필요한 내부 문서를 정리합니다.
- `cycle-report`: 한 사이클의 증거를 HTML 보고서로 고정합니다.

사용자는 이 역할들을 직접 외울 필요는 없습니다.
다만 한 사이클이 여러 역할을 거쳐 검수된다는 점만 이해하면 됩니다.

## 보고서 다시 열기

이미 생성된 보고서는 아래 명령으로 다시 열 수 있습니다.

```bash
cargo run -p xavi-dev-console -- open-cycle --cycle-id <cycle-id>
```

## 가장 중요한 사용 원칙

1. 큰 작업을 한 번에 맡기지 않습니다.
2. 항상 작은 `1회 작업 사이클`로 나눕니다.
3. 구현 전에 계획을 확인합니다.
4. 구현 뒤에는 HTML 보고서를 확인합니다.
5. 다음 지시는 직전 보고서를 보고 냅니다.

이 저장소는 완성 앱이 아니라,
AI와 개발을 검수 가능한 작은 단위로 관리하기 위한 부트스트랩입니다.

## 주의

- `.xavi/` 안에는 로컬 작업 증거와 보고서가 남습니다.
- 비밀 키, 토큰, 비밀번호 같은 민감한 정보는 프롬프트에 넣지 않는 것이 좋습니다.
- `.xavi/` 를 직접 공유, 커밋, 업로드하면 작업 증거가 노출될 수 있습니다.
