# Inject Subject Once

이 문서는 이 루트 프로젝트를 기반으로 **새 프로젝트 주제를 처음 1회 주입할 때만** 실행하는 일회성 주제 주입 프로토콜이다.

사용자는 보통 아래처럼 이 문서를 실행한다.

```text
이제 프로젝트 주제를 정할꺼야, inject_subject_once.md 읽고 다음 내 주제에 맞게 기본 틀을 작성하고, 내용을 추가해줘.
```

하지만 사용자가 반드시 이 파일명을 알아야 하는 것은 아니다.
최상위 `orchestra` 가 `starter.md` 를 읽은 뒤, 아직 범용 health-check 부트스트랩 상태인 저장소에서 사용자가 새 프로젝트 주제나 제품 아이디어를 말하면, 그것은 곧 이 문서의 실행 트리거다.
즉 사용자가 `inject_subject_once.md` 를 직접 언급하지 않아도 `orchestra` 는 이 문서를 찾아 읽고 적용해야 한다.

이 문서는 기본적으로 최상위 `orchestra` 세션이 읽고 실행한다.
`orchestra` 는 이 문서만으로 필요한 작업을 이해한 뒤, 실제 분석, 계획, 구현, 검수, 테스트, 사용자 문서 작업을 역할별 서브 에이전트에 나누어 맡겨야 한다.
한 에이전트가 모든 역할을 직접 수행하는 방식으로 해석하면 안 된다.

## 1. 이 문서의 목적

이 문서는 새 프로젝트 주제가 정해지는 순간, 현재의 제네럴한 기본 틀을 **프로젝트 주제에 맞는 실제 초기 구조**로 바꾸기 위한 규약이다.

핵심 목적은 네 가지다.

- 현재의 범용 템플릿을 주제 맞춤형 기본 틀로 확장한다.
- 문서 시스템에 프로젝트 주제 내용을 주입한다.
- 클린 아키텍처와 하네스 중심 개발 구조를 주제에 맞게 구체화한다.
- 역할별 에이전트 문서와 사람용 문서를 주제에 맞게 채운다.

## 2. 이 문서가 절대 훼손하면 안 되는 기본 원칙

아래 원칙은 **항상 반드시 유지**해야 한다.
프로젝트 주제가 무엇이든 이 원칙들은 수정하거나 제거하면 안 된다.

### 코드 구조 원칙

- Rust 워크스페이스 구조를 유지한다.
- 클린 아키텍처 레이어 분리를 유지한다.
- 최소한 아래 레이어 축은 유지한다.
  - `crates/xavi-domain`
  - `crates/xavi-application`
  - `crates/xavi-infrastructure`
  - `crates/xavi-harness`
  - `apps/xavi-bootstrap`
- 안쪽 레이어가 바깥 레이어를 참조하지 않는 방향을 유지한다.
- 하네스는 별도 outer-layer 검증 시스템으로 유지한다.

### 에이전트 운영 원칙

- 총 9개의 세션 역할을 유지한다.
  - `orchestra`
  - `planning`
  - `analysis`
  - `codegen`
  - `review`
  - `test`
  - `ai-docs`
  - `ephemeral`
  - `user-docs`
- 각 역할은 자기 역할에만 충실해야 한다.
- 각 역할은 다른 역할에 침범하면 안 된다.
- 역할별 문서 접근 경계를 유지한다.
- 서브 에이전트는 사용 가능한 최신 프론티어급 모델을 기본으로 사용한다. 2026-05-22 현재 기준 기본값은 `gpt-5.5` 와 `xhigh` 다.
- 개발 업무와 주제 주입 구현은 예시가 아니라 절대 고정 루프를 따른다. `planning` 뒤와 `codegen` 앞에는 사용자 보고/수정 게이트를 둔다. 역할 실행 순서는 `planning` -> `codegen` -> `review` -> `codegen` -> `test` -> `analysis` -> `codegen` -> `review` -> `codegen` -> `test` -> `analysis` -> `codegen` -> `test` -> `planning` 보고.
- 개발 업무 기본 7개 서브 에이전트는 `planning`, `codegen`, `review`, `test`, `analysis`, `user-docs`, `ai-docs` 다.

### 역할별 핵심 고정 원칙

- `orchestra`:
  - 최상위 기본 역할이다.
  - 사용자와 직접 대화하고 현재 상황을 함께 파악한다.
  - 계획 작성도 직접 하지 않고 `planning` 서브 에이전트에 맡긴다.
  - 서브 에이전트 생성, 종료, 재생성, 반환 검수를 맡는다.
- `planning`:
  - `orchestra` 가 넘긴 사용자 의도와 구현 목표를 바탕으로 계획 초안을 만든다.
  - 개발 루프 종료 후 처음 계획 대비 완성도와 남은 문제 보고서를 만든다.
  - 사용자와 직접 장기 대화를 주도하지 않는다.
- `analysis`:
  - 테스트 문제점 리스트를 읽기 전용으로 분석하고 근본 원인을 정리한다.
  - 코드나 문서를 직접 수정하지 않는다.
- `codegen`:
  - 코드 생성만 한다.
  - 정상 작업에서는 문서 작업을 하지 않는다. 단, 중간 중단 인계 종료의 자기 handoff 는 예외다.
- `review`:
  - 생성된 코드 검수와 하네스 보강을 맡는다.
  - 제품 코드 수정은 `orchestra` 를 통해 `codegen` 으로 되돌린다.
  - 검수는 **반드시 하네스를 우선 기준으로** 수행한다.
  - 자기 검수 문서를 스스로 관리한다.
- `test`:
  - 검증 명령 실행, 문제점 리스트 작성, 재테스트를 맡는다.
  - 제품 코드 수정과 실패 근본 원인 분석은 직접 하지 않는다.
- `ephemeral`:
  - 임시 조사/보조 작업을 맡는다.
  - 자기 임무와 수행 내역, 남은 일을 문서화한다.
- `ai-docs`:
  - AI 에이전트들이 개발을 위해 참조하는 내부 문서 시스템을 전담한다.
  - `docs/agent/` 와 루트 운영 문서만 관리하고 사용자용 문서는 작성하지 않는다.
  - 1회 개발 사이클마다 역할별 산출물, 사용자 반응, 다음 사이클 인계 정보를 AI 전용 문서로 갱신한다.
- `user-docs`:
  - 사용자에게 보여줄 문서를 전담한다.
  - 코드와 문서 변화를 읽고 사람 기준으로 쉽게 풀어서 문서화한다.
  - 이 역할은 반드시 유지해야 하며 축소/병합/삭제하면 안 된다.

### 문서 시스템 원칙

- 사람용 문서와 AI용 문서는 분리한다.
- 내부 AI 문서 시스템은 역할별로 분리한다.
- 사용자 문서 전담 AI 는 정상 작업에서 `README.md` 와 `docs/human/` 을 관리한다. 기본 작업 공간은 `docs/human/user-docs/` 이며, 프로젝트 주제 주입이나 공개 문서 갱신처럼 명시된 경우에는 `docs/human/` 일반 문서도 관리한다. 중간 중단 인계 종료에서는 자기 인계만 `docs/agent/user-docs/handoff/latest.md` 에 남긴다.
- 사용자 문서 전담 AI 는 반드시 `docs/agent/user-docs/README.md` 를 먼저 읽고 역할 경계를 확정한 뒤, 정상 사용자 문서 작업일 때 `README.md` 또는 `docs/human/` 의 해당 문서로 이동한다.
- AI 문서 전담 AI 는 `docs/agent/` 와 루트 운영 문서만 관리한다.
- 사용자용 문서와 AI 에이전트용 문서는 독자, 위치, 문체, 관리 역할을 완전히 분리한다.
- `starter.md` 는 최상위 `orchestra` 세션의 부팅 규약 문서로 유지한다.
- 서브 에이전트에는 `starter.md` 를 반복해서 읽히지 않고 각 역할의 `docs/agent/<role>/README.md` 경로만 전달한다.

## 3. 이 문서가 해야 하는 일

이 문서를 읽은 `orchestra` 세션은 사용자가 주제를 제공하면 아래 작업을 역할별 서브 에이전트 흐름으로 조율해야 한다.

1. 사용자와 직접 대화해 프로젝트 주제를 해석한다.
2. 개발 업무 기본 7개 서브 에이전트 `planning`, `codegen`, `review`, `test`, `analysis`, `user-docs`, `ai-docs` 를 준비한다.
3. `planning` 서브 에이전트에 핵심 도메인 개념, 범위, 성공 기준, 미결 사항을 계획 초안으로 정리하게 한다.
4. 이후 구현은 `starter.md` 의 개발 업무 절대 고정 루프 순서로만 지시한다.
5. 역할별 AI 문서 시스템의 주제 정보 갱신은 `ai-docs` 역할에 맡긴다.
6. 사용자용 문서 시스템은 `user-docs` 역할에 맡기고, AI 에이전트용 문서 시스템은 `ai-docs` 역할에 맡긴다.

## 3-1. 주제 주입은 부분 수정이 아니다

이 문서를 실행한 작업은 단순히 `README.md` 한두 문장이나 예제 코드 이름만 바꾸는 작업이 아니다.
반드시 저장소 전체가 "범용 부트스트랩" 에서 "사용자 주제의 초기 프로젝트" 로 읽히도록 바뀌어야 한다.

`orchestra` 는 주제 주입 작업을 아래 범위로 이해해야 한다.

- 루트 README 와 사람용 문서는 사용자 주제를 기준으로 다시 읽혀야 한다.
- 내부 AI 역할 문서는 다음 개발 사이클에서 같은 주제를 이어받을 수 있어야 한다.
- 코드에는 주제와 직접 연결되는 첫 domain/application/infrastructure/bootstrap 흐름이 보여야 한다.
- 하네스에는 주제와 직접 연결되는 첫 fixture/double/scenario/assertion/test 흐름이 있어야 한다.
- health-check 예제는 의미가 있으면 보조 상태 확인으로 남길 수 있지만, 프로젝트의 중심 설명이나 유일한 코드 흐름으로 남겨두면 안 된다.
- 마지막 검수에서는 `xavi_1`, generic health-check, bootstrap template 설명이 사용자 주제 설명을 밀어내고 있지 않은지 확인해야 한다.

## 4. 사용자가 주제를 아직 충분히 주지 않았을 때

사용자가 주제를 명확히 주지 않았으면, 작업을 시작하기 전에 짧게 필요한 정보를 요청한다.
질문은 길게 하지 말고, 아래 정보를 확보할 수 있을 정도로만 묻는다.

- 프로젝트가 해결하려는 문제
- 주요 사용자 또는 사용 시나리오
- 핵심 기능 3~5개
- 외부 시스템 연동 필요 여부
- CLI / server / desktop / batch 등 실행 형태

정보가 일부 부족하더라도 합리적인 최소 가정으로 시작할 수 있으면 시작한다.
단, 구조를 잘못 고정할 위험이 크면 사용자에게 짧게 재확인한다.

## 5. 프로젝트 주제 주입 시 허용되는 변경

주제 주입 시 아래 변경은 허용된다.

- `README.md` 를 주제에 맞게 재작성 또는 확장
- `starter.md` 에 주제 관련 참고 문맥 추가
- `docs/human/` 에 주제 설명 문서 추가
- `docs/human/user-docs/` 에 사용자용 초기 주제 문서 추가
- `docs/agent/*` 역할 문서에 주제 맞춤형 초기 내용 추가
- domain/application/infrastructure/harness 내부에 주제용 모듈 추가
- 현재의 generic 예시 코드(`health` 같은 placeholder)를 교체하거나 확장
- 하네스 fixture/scenario/assertion/double 구조를 주제의 첫 검증 흐름에 맞게 확장

## 6. 프로젝트 주제 주입 시 금지되는 변경

아래 변경은 금지된다.

- 클린 아키텍처 레이어 분리 제거
- `xavi-harness` 제거
- `review` 의 하네스 우선 검증 원칙 제거
- 9개 세션 역할 제거 또는 병합
- `orchestra` 최상위 역할 제거
- `planning` 을 최상위 역할로 되돌리거나 planning 을 서브 에이전트가 아닌 메인 역할로 병합
- `planning` 뒤 사용자 보고/수정 게이트와 `planning` -> `codegen` -> `review` -> `codegen` -> `test` -> `analysis` -> `codegen` -> `review` -> `codegen` -> `test` -> `analysis` -> `codegen` -> `test` -> `planning` 보고 절대 고정 루프 제거
- 최신 프론티어급 기본 모델 정책 제거
- `user-docs` 역할 제거 또는 내부 AI 문서 시스템으로 이동
- `ai-docs` 역할 제거 또는 사용자용 문서 시스템으로 이동
- `codegen` 에 정상 문서화 책임 부여. 단, 중간 중단 인계 종료의 자기 handoff 예외는 유지
- 역할 간 문서 경계 제거
- 사람용 문서와 AI용 문서 통합
- `starter.md` 의 역할 부팅 규약 제거

## 7. 반드시 수정하거나 추가해야 하는 산출물

프로젝트 주제 주입 시 최소한 아래 산출물은 반드시 만들어야 한다.
사용자나 공개 독자가 읽는 `README.md` 와 `docs/human/` 문서는 `user-docs` 역할이 맡고, AI 운영 규칙인 `starter.md`, `ender.md`, `inject_subject_once.md`, `docs/agent/` 는 `ai-docs` 역할이 맡는다.

### 루트

- `README.md`
  - 프로젝트 주제 요약
  - 핵심 기능
  - 레이어 개요
  - 실행/검증 시작점
- `starter.md`
  - 기본 규약은 유지
  - 필요하면 프로젝트 주제 관련 예시나 현재 주제 스냅샷만 추가
  - 역할 규칙과 기본 원칙은 절대 약화하지 않음

### 사람용 문서

- `docs/human/README.md`
  - 현재 프로젝트 주제에 맞는 설명을 추가
- 새 파일 권장:
  - `docs/human/project-subject.md`
  - `docs/human/architecture-subject-map.md`

### 사용자 문서 전담 시스템

- `docs/human/user-docs/README.md`
  - 현재 프로젝트 주제 기준으로 읽히게 보완 가능
- 새 파일 권장:
  - `docs/human/user-docs/project-overview.md`
  - `docs/human/user-docs/initial-report.md`

### 내부 AI 역할 문서

- `docs/agent/orchestra/README.md`
  - 최상위 사용자 대화, 서브 에이전트 관리, 반환 검수 기준이 보존되어야 한다.
- `docs/agent/analysis/README.md`
  - 주제 기준 분석 범위와 위험 확인 기준이 보존되어야 한다.
- `docs/agent/codegen/backlog.md`
  - 주제 기준 초기 구현 backlog 로 채운다.

  - 주제 기준 검수 범위 초기값을 넣는다.
- `docs/agent/review/checklist.md`
  - 주제 기준 검수 체크포인트를 보강한다.
- `docs/agent/test/README.md`
  - 주제 기준 실행해야 할 기본 검증 명령과 문제점 리스트 작성 기준이 보존되어야 한다.
- `docs/agent/planning/brief.md`
  - 주제 정의, 목표, 범위, 제외 범위, 성공 기준을 채운다.
- `docs/agent/planning/decision-log.md`
  - 초기 결정과 미결 사항을 남긴다.
- `docs/agent/planning/dialogue-log.md`
  - 주제 주입 시점의 사용자 의도를 정리한다.
- `docs/agent/ai-docs/README.md`
  - AI 에이전트용 문서 갱신과 인계 정보 관리 기준이 보존되어야 한다.
- `docs/agent/user-docs/README.md`
  - 사용자 문서 전담 역할의 진입 경로와 `README.md`, `docs/human/` 전용 갱신 경계가 보존되어야 한다.
- `docs/agent/ephemeral/`
  - 필요하면 주제 탐색용 초기 세션 템플릿이나 예시 세션을 추가한다.

## 8. 코드 구조 주입 규칙

주제가 정해지면 현재의 generic 예제를 그대로 두기만 하지 말고, 실제 주제에 맞는 첫 구조를 코드에 넣어야 한다.

기본 방향은 아래와 같다.

### `crates/xavi-domain`

- 주제의 핵심 엔티티, 값 객체, 상태, 규칙을 둔다.
- 현재 generic 모델이 의미 없으면 주제용 모델로 교체하거나 별도 모듈로 확장한다.

### `crates/xavi-application`

- 주제의 첫 유스케이스와 포트를 둔다.
- 최소한 사용자 주제와 직접 연결되는 첫 시나리오 하나는 구조로 드러나야 한다.

### `crates/xavi-infrastructure`

- 주제에서 필요한 첫 외부 어댑터 틀을 둔다.
- 실제 외부 시스템이 아직 없어도 in-memory 또는 fake 성격의 기본 어댑터를 준비할 수 있다.

### `crates/xavi-harness`

- 주제 기준 첫 검증 흐름을 담는 fixture/scenario/assertion/double 구조를 만든다.
- 이 하네스는 이후 `review` 역할이 **반드시** 사용하게 될 기본 검증 진입점이 된다.

### `apps/xavi-bootstrap`

- 주제 기준 첫 composition root 를 보여준다.
- 최소한 현재 주제의 첫 유스케이스가 어떻게 조립되는지 드러나야 한다.

## 9. 하네스 주입 규칙

프로젝트 주제를 주입할 때, 하네스는 generic 구조만 남겨두지 말고 주제에 맞는 첫 검증 흐름을 추가해야 한다.

최소 요구사항:

- 주제와 직접 연결되는 첫 scenario 추가
- 그 scenario 에 필요한 fixture 추가
- 필요한 harness-owned double 추가
- 필요한 assertion helper 추가
- 최소 1개 이상의 하네스 기반 테스트 추가

즉 주제 주입이 끝난 시점에는, 하네스가 단순 틀만 있는 상태가 아니라 **그 프로젝트 주제를 실제로 검증하기 시작할 수 있는 상태**여야 한다.

## 10. 역할별 주제 주입 원칙

### `analysis`

- 테스트 문제점 리스트를 읽기 전용으로 분석하고 근본 원인을 정리한다.
- 주제 기준 테스트 실패가 있으면 수정 후보와 다음 테스트 확인 지점을 `orchestra` 에 반환할 수 있어야 한다.

### `orchestra`

- 주제 주입 전반을 사용자와 함께 조율한다.
- 계획 작성은 직접 하지 않고 `planning` 서브 에이전트에 맡긴다.
- 각 서브 에이전트의 반환 요약과 `Context Report` 를 검수한다.

### `codegen`

- 정상 문서화 책임을 추가하지 않는다. 중간 중단 인계 종료의 자기 handoff 예외는 유지한다.
- 주제에 맞는 구현 backlog, constraints 는 `orchestra` 가 취합한 입력을 바탕으로 `ai-docs` 가 미리 준비하고, `codegen` 은 이를 직접 작성하지 않는다.

### `review`

- 하네스 기반 검증 책임은 그대로 유지한다.
- 주제 기준 검수 범위, 첫 검수 시나리오, 회귀 포인트가 문서에 드러나야 한다.
- 제품 코드 수정은 직접 확장하지 않고 `codegen` 으로 되돌리는 기준이 드러나야 한다.

### `test`

- 주제 기준 기본 검증 명령과 재테스트 기준이 드러나야 한다.
- 실패 명령, 핵심 오류, 문제점 리스트를 남길 수 있어야 한다.
- 실패의 근본 원인 분석은 `analysis` 에 맡긴다.

### `planning`

- `orchestra` 가 전달한 사용자의 주제 설명을 brief 와 decision log 로 정리해야 한다.
- 이 문서 주입 시점부터 planning 문서는 비어 있으면 안 된다.
- 구현 전 사용자 보고에 쓸 계획 초안과 계획 수정안을 반환해야 한다.

### `ephemeral`

- 즉시 필요하진 않더라도, 주제 탐색이나 외부 조사에 쓸 수 있는 초기 세션 규칙을 유지한다.

### `ai-docs`

- 1회 사이클 전체 요약, 역할별 산출물, 사용자 반응, 다음 사이클 인계 정보를 AI 에이전트 전용 문서로 갱신한다.
- 사용자용 문서는 작성하지 않는다.
- `user-docs` 와 같은 내용을 섞지 않는다.

### `user-docs`

- 이 역할은 특히 중요하다.
- 주제가 주입되면 사용자가 읽을 수 있는 프로젝트 개요 문서가 바로 생겨야 한다.
- 코드가 아직 적더라도 "이 프로젝트가 무엇이고, 어떤 구조로 시작했고, 앞으로 어떤 코드가 어떤 자리에 생길지" 를 사용자 관점으로 설명하는 초기 문서를 반드시 만든다.

## 11. 수행 순서

이 문서를 읽은 `orchestra` 세션은 실제 작업을 아래 순서로 조율한다.

1. 사용자의 프로젝트 주제를 해석한다.
2. 훼손하면 안 되는 기본 원칙을 먼저 고정한다.
3. 기본 7개 서브 에이전트 `planning`, `codegen`, `review`, `test`, `analysis`, `user-docs`, `ai-docs` 를 준비한다.
4. `planning` 서브 에이전트로 주제의 핵심 도메인 개념, 첫 유스케이스, 구현 계획 초안을 받는다.
5. `orchestra` 는 처음 계획을 사용자에게 보고하고, 사용자의 수정 의견이나 진행 승인을 받는다.
6. 사용자가 계획 수정을 요구하면 `planning` 에 다시 전달해 계획 개정안을 받는다.
7. `codegen` 이 승인되었거나 개정된 계획을 바탕으로 코드 구조에 주제용 첫 모듈과 첫 흐름을 추가한다.
8. `review` 가 생성 코드를 검수하고 문제점 리스트를 만든다.
9. `codegen` 이 검수자 문제점 리스트를 반영해 수정한다.
10. `test` 가 1차 테스트를 실행한다. 완전 성공이면 문제 분석/재수정 루프를 건너뛰고 17단계의 결과 취합과 최종 보고 단계로 넘어간다.
11. 1차 테스트 문제가 있으면 `analysis` 가 근본 원인을 분석하고 `codegen` 이 수정한다.
12. `review` 가 수정된 부분만 재검수하고 `codegen` 이 재검수 문제점 리스트를 수정한다.
13. `test` 가 2차 테스트를 실행한다.
14. 2차 테스트 문제가 있으면 `analysis` 가 근본 원인을 분석하고 `codegen` 이 수정한다.
15. `test` 가 3차 테스트를 실행한다.
16. 3차 테스트 이후에는 더 이상 코드 개발을 진행하지 않는다.
17. 모든 결과를 취합해 `planning` 에 보내 최종 완성도 보고서를 받는다.
18. `orchestra` 가 `planning` 보고서를 사용자에게 전달하고 사용자 반응을 받는다.
19. `user-docs` 서브 에이전트가 1회 사이클 전체 요약, `planning` 보고서, 사용자 반응, 실제 변경과 확인된 상태를 사용자 문서 전담 영역에 기록한다.
20. `ai-docs` 서브 에이전트가 1회 사이클 전체 요약, 역할별 산출물, 문제점, 근본 원인, 사용자 반응, 다음 사이클 인계 정보를 AI 에이전트 전용 문서에 기록한다.
21. 모든 서브 에이전트의 반환 요약과 `Context Report` 를 `orchestra` 가 검수한다.
22. 결과적으로 이 저장소가 "제네럴 템플릿" 이 아니라 "해당 주제의 초기 프로젝트" 로 읽히는지 확인한다.

## 12. 완료 조건

주제 주입은 아래 조건이 충족되면 완료로 본다.

- 사용자가 준 주제가 문서에 반영되어 있다.
- 사람용 문서에 주제 설명이 있다.
- 역할별 AI 문서에 주제 맞춤형 초기 내용이 있다.
- `ai-docs` 가 AI 에이전트용 인계 정보와 다음 사이클 제약을 반영했다.
- 코드 구조에 주제용 첫 흐름이 있다.
- 하네스에 주제용 첫 검증 흐름이 있다.
- 사용자용 문서에 초기 프로젝트 설명이 있다.
- 기본 원칙은 하나도 훼손되지 않았다.

## 13. 이 문서는 한 번만 실행한다

이 문서는 원칙적으로 프로젝트 시작 시 딱 한 번만 실행한다.

이미 주제가 주입된 뒤에는 이 문서를 다시 실행하지 않는다.
다시 실행하려면 사용자가 명시적으로 재부트스트랩 또는 전면 재설계를 요청해야 한다.

만약 이미 주입된 흔적이 보이면, 즉시 덮어쓰지 말고 사용자에게 재실행 의도인지 짧게 확인한다.

## 14. 한 줄 요약

이 문서는 현재의 제네럴한 개발 시스템 틀을, **기본 철학은 절대 훼손하지 않은 채**, 사용자가 정한 실제 프로젝트 주제에 맞는 첫 코드/문서/하네스/역할별 운영 틀로 바꾸기 위한 1회성 주제 주입 규약이다.
