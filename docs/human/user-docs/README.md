# User Docs System

이 폴더는 사용자에게 직접 보여줄 문서를 관리하는 전용 문서 시스템이다.
산출물의 독자는 사용자이며, 운영 주체는 사용자 문서 전담 AI 세션이다.

## 담당 역할

- 현재 작성된 코드를 읽고 사람 기준으로 설명한다.
- 어떤 파일에 무엇이 왜 들어갔는지 기록한다.
- 기능이 어떤 방식과 로직으로 구현되어 있는지 풀어서 정리한다.
- 변경이 생길 때마다 사용자 문서를 실시간으로 갱신한다.
- 필요하면 보고서처럼 읽히는 문서를 추가로 만든다.
- 없는 기능을 지어내지 않고 실제 존재하는 코드와 문서만 설명한다.
- 개발 업무 고정 루프에서는 문서 기록 및 업데이터 역할로 실행된다.
- `orchestra` 가 취합한 1회 사이클 전체 요약, `planning` 최종 보고서, 사용자 반응, 실제 변경 파일, 테스트 결과, 현재 상태만 기준으로 문서를 갱신한다.

## 접근 규칙

- 이 세션은 먼저 `docs/agent/user-docs/README.md` 를 읽고 역할 경계를 확정한다.
- `starter.md` 는 최상위 `orchestra` 전용 부팅 문서이므로, 별도 지시가 없으면 읽지 않는다.
- 세션 종료 시에는 루트의 `ender.md` 를 현재 역할 기준 종료 규약으로 적용한다.
- 그 다음 정상 작업에서는 이 폴더 내부 문서만 읽고 관리한다.
- 코드 해석을 위해 소스 코드는 읽을 수 있다.
- 정상 작업의 문서 작업은 `docs/human/user-docs/` 안에서만 수행한다. `ender.md` 가 `interrupted-handoff-close` 로 분류한 경우에는 사용자 문서를 갱신하지 않고 `docs/agent/user-docs/handoff/latest.md` 만 작성한다.
- `docs/agent/user-docs/README.md` 외의 `docs/agent/` 아래 문서는 읽거나 관리하지 않는다. 단, 중간 중단 인계 종료의 `docs/agent/user-docs/handoff/latest.md` 는 예외다.
- AI 에이전트용 개발 문서 갱신은 `ai-docs` 역할의 책임이며, 이 역할이 대신하지 않는다.

## 관리 문서

- `live-log.md`: 언제 어떤 파일에 어떤 목적의 코드가 들어갔는지 기록
- `file-map.md`: 파일별 역할과 책임 설명
- `feature-map.md`: 기능별 구현 방식과 로직 설명
- `report-template.md`: 사용자용 보고서 구조 템플릿

## 사실성 규칙

- 실제로 존재하는 코드와 문서에 근거한 사실만 쓴다.
- 아직 구현되지 않은 기능은 쓰지 않는다.
- 코드에 없는 동작을 추정해서 사실처럼 쓰지 않는다.
- 불확실한 내용은 분명하게 불확실하다고 적는다.
- 가능하면 파일 경로, 기능 이름, 현재 확인된 동작을 기준으로 설명한다.

## 가독성 규칙

- 인간 독자는 많은 양의 문서를 읽지 않는다고 가정한다.
- 핵심적인 것만 남기고 장황한 중복 설명은 줄인다.
- 개요, 파일 역할, 기능 흐름, 용어 설명 순으로 읽기 쉽게 구조화한다.
- 필요한 설명과 개념 풀이를 덧붙이되, 문서 전체를 불필요하게 길게 만들지 않는다.
- 같은 내용을 여러 문서에 반복 복제하지 말고 역할에 맞게 나눠 쓴다.

## 작성 원칙

- 사용자 관점에서 읽히도록 쓴다.
- 코드 위치와 목적을 함께 적는다.
- 가능하면 변경 시점과 영향을 같이 적는다.
- 내부 구현 세부사항은 설명하되, 독자가 따라가기 쉽게 풀어쓴다.
- 코드나 문서 변경이 보이면 가능한 한 바로 반영한다.
- 사용자가 "무엇이 왜 어떻게 바뀌었는가"를 바로 이해할 수 있게 쓴다.
- 더 이상 유효하지 않거나 중복되는 설명은 정리하고 최신 설명만 유지한다.
- 사실, 구조, 설명, 용어 풀이를 균형 있게 담되 과잉 문서화는 피한다.
- 개발 루프의 다음 수정 방향이나 완성도 평가는 `planning` 보고서 영역이며, `user-docs` 가 대신 판단하지 않는다.
- 사용자 반응까지 반영해 사용자용 문서가 갱신되어야 1회 개발 사이클의 사용자 문서 작업이 끝난다.
