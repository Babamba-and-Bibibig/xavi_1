# Human Docs

이 문서는 사람 개발자와 리뷰어가 이 프로젝트의 구조를 빠르게 이해할 수 있도록 정리한 안내서다.
특히 "어떤 폴더가 왜 존재하는가", "어디에 무엇을 넣어야 하는가", "레이어 경계를 어떻게 지켜야 하는가"를 명확하게 설명하는 것을 목표로 한다.

프로젝트는 Rust 워크스페이스 기반으로 구성되어 있으며, 클린 아키텍처를 중심 원칙으로 삼는다.
즉, 핵심 비즈니스 규칙은 바깥 기술 세부사항에 오염되지 않도록 안쪽 레이어에 두고, 실행 환경과 외부 시스템 연동은 바깥 레이어로 밀어낸다.

## 빠른 이동

- 프로젝트 구조 한눈에 보기
- 루트 폴더의 역할
- `apps/` 하위 폴더 설명
- `crates/` 하위 폴더 설명
- `docs/` 하위 폴더 설명
- 루트 주요 파일 설명
- 의존 방향 정리
- 폴더를 사용할 때의 실무 기준

## 프로젝트 구조 한눈에 보기

```text
xavi_1/
├── .cargo/
├── apps/
│   └── xavi-bootstrap/
├── crates/
│   ├── xavi-domain/
│   ├── xavi-application/
│   ├── xavi-infrastructure/
│   └── xavi-harness/
├── docs/
│   ├── human/
│   │   └── user-docs/
│   └── agent/
│       ├── codegen/
│       ├── review/
│       ├── planning/
│       └── ephemeral/
├── Cargo.toml
├── README.md
├── .gitignore
├── rustfmt.toml
└── starter.md
```

이 구조는 크게 네 축으로 나뉜다.

- 실행 가능한 진입점은 `apps/` 아래에 둔다.
- 재사용 가능한 핵심 로직은 `crates/` 아래의 레이어별 라이브러리로 둔다.
- 문서는 `docs/` 아래에 사람용과 에이전트용으로 분리한다.
- 공통 빌드 규칙과 포맷 규칙은 루트 설정 파일에 둔다.

## 루트 폴더의 역할

### `.cargo/`

이 폴더는 Cargo 로컬 설정을 두는 곳이다.
프로젝트 전체에 공통으로 적용할 명령 별칭이나 개발 편의 설정을 관리한다.

현재 이 프로젝트에서는 `.cargo/config.toml` 에 워크스페이스 전체를 대상으로 하는 명령 별칭이 들어 있다.
예를 들어 `cargo xcheck`, `cargo xtest`, `cargo xclippy` 같은 형태로 자주 쓰는 검증 명령을 짧게 실행할 수 있게 만든다.

이 폴더의 역할은 "비즈니스 로직을 담는 것"이 아니라 "개발 경험을 정리하는 것"이다.
따라서 여기에는 코드가 아니라 도구 설정만 두는 것이 맞다.

### `apps/`

`apps/` 는 실제로 실행 가능한 애플리케이션의 진입점을 두는 폴더다.
클린 아키텍처에서 말하는 composition root 를 두기에 적합한 위치이기도 하다.

이 폴더에는 최종적으로 다음과 같은 것들이 들어갈 수 있다.

- CLI 실행 파일
- 서버 바이너리
- 배치 잡 실행기
- 운영용 어드민 도구

핵심 원칙은, `apps/` 는 "무언가를 실행하는 곳"이지 "핵심 규칙을 정의하는 곳"이 아니라는 점이다.
즉 이 레이어에서는 의존성을 조립하고, 설정을 읽고, 어떤 인프라 구현을 어떤 유스케이스에 연결할지 결정하는 역할을 맡는다.

### `crates/`

`crates/` 는 프로젝트의 핵심 라이브러리들을 모아두는 공간이다.
현재 구조에서는 클린 아키텍처 레이어를 crate 단위로 강하게 분리했다.

이렇게 분리하면 다음 장점이 있다.

- 의존 방향을 Cargo 차원에서 강제하기 쉽다.
- 레이어 침범이 발생하면 `Cargo.toml` 의존 관계에서 바로 드러난다.
- 테스트와 리팩터링 범위를 구분하기 쉽다.
- 장기적으로 팀과 에이전트가 같은 기준으로 파일 위치를 판단하기 쉬워진다.

즉 `crates/` 는 단순한 폴더가 아니라, 프로젝트의 설계 경계를 물리적으로 드러내는 핵심 구조다.

### `docs/`

`docs/` 는 프로젝트 문서 시스템 전체를 담는 곳이다.
이 프로젝트의 특징은 문서를 하나로 합치지 않고, 독자를 기준으로 두 갈래로 나누고, AI 문서는 다시 역할별로 분해했다는 점이다.

- `docs/human/`: 사람 개발자와 리뷰어를 위한 문서
- `docs/agent/`: AI 에이전트 문서 시스템 루트

그리고 `docs/agent/` 내부에는 다시 네 개의 독립 문서 시스템이 있다.

- `docs/agent/codegen/`: 코드 생성 전용
- `docs/agent/review/`: 수정 검수 전용
- `docs/agent/planning/`: 기획 대화 전용
- `docs/agent/ephemeral/`: 임시 서브 AI 전용

또한 `docs/human/` 아래에는 사용자에게 직접 보여줄 문서를 정리하는 특수 구역을 둘 수 있다.

- `docs/human/user-docs/`: 사용자 문서 전담 AI 가 관리하는 사용자용 실시간 문서

같은 주제를 다루더라도 두 문서는 밀도와 목적이 다를 수 있다.
사람용 문서는 배경, 이유, 설계 맥락을 충분히 담고, 에이전트용 문서는 역할별 규칙, 체크리스트, 작업 절차를 짧고 직접적으로 담는 식으로 운영하는 것이 적절하다.

## `apps/` 하위 폴더 설명

### `apps/xavi-bootstrap/`

현재 워크스페이스에서 실제 실행 진입점 역할을 맡는 바이너리 크레이트다.
이 이름에 들어 있는 `bootstrap` 은 단순한 시작 프로그램이라는 뜻이 아니라, 애플리케이션을 조립하는 바깥쪽 루트라는 의미를 가진다.

이 폴더에서 담당해야 하는 일은 다음과 같다.

- 프로그램 시작
- 환경 설정 로딩
- 로거 초기화
- 인프라 어댑터 생성
- 애플리케이션 서비스와 어댑터 연결
- 최종 실행 흐름 시작

반대로 이 폴더에 두면 안 되는 것은 다음과 같다.

- 핵심 비즈니스 규칙
- 도메인 모델 정의
- 외부 기술과 무관해야 하는 유스케이스 정책

쉽게 말하면 `apps/xavi-bootstrap/` 은 "시스템을 켜는 곳"이지 "시스템의 본질을 정의하는 곳"이 아니다.

## `crates/` 하위 폴더 설명

### `crates/xavi-domain/`

이 폴더는 가장 안쪽 레이어인 도메인 레이어다.
프로젝트에서 가장 오래 살아남아야 하는 핵심 개념과 규칙이 여기에 들어간다.

도메인 레이어에 들어갈 수 있는 대표 요소는 다음과 같다.

- 엔티티
- 값 객체
- 도메인 서비스
- 도메인 규칙
- 불변 조건
- 상태 전이 규칙

이 레이어의 가장 중요한 제약은, 바깥 세상에 대해 몰라야 한다는 점이다.
예를 들어 HTTP, 데이터베이스, 파일 시스템, 프레임워크 타입, 런타임 세부사항에 의존하면 안 된다.

현재는 예시로 `health.rs` 가 들어 있으며, 시스템 상태를 나타내는 `HealthStatus` 와 `HealthReport` 같은 순수 도메인 모델이 정의되어 있다.
이것은 구조를 보여주기 위한 시작점이고, 이후 실제 프로젝트 개념이 구체화되면 도메인 중심 모델로 확장해 나가면 된다.

### `crates/xavi-application/`

이 폴더는 애플리케이션 레이어다.
도메인 레이어를 사용해서 "무엇을 하게 할 것인가"를 정의하는 유스케이스가 이곳에 들어간다.

애플리케이션 레이어는 보통 다음 역할을 맡는다.

- 유스케이스 조합
- 입력과 출력 흐름 조정
- 트랜잭션 경계 정의
- 포트 인터페이스 정의
- 도메인 호출 순서 제어

이 레이어는 도메인을 사용하지만, 인프라 구현 세부사항은 직접 가지지 않는다.
대신 포트 인터페이스를 통해 "무엇이 필요하다"만 선언한다.

현재 구조에서:

- `ports/` 는 outbound port 를 두는 공간이다.
- `services/` 는 실제 유스케이스를 담는 공간이다.

예를 들어 `health_status_reader.rs` 는 바깥에서 상태를 읽어오는 계약이고, `health_check_service.rs` 는 그 계약을 사용해 건강 상태를 조회하는 유스케이스다.

즉 애플리케이션 레이어는 "기능 시나리오의 중심"이라고 보면 된다.

### `crates/xavi-infrastructure/`

이 폴더는 인프라 레이어다.
외부 세계와 실제로 연결되는 구현체를 두는 곳이다.

대표적으로 다음 요소들이 이 레이어에 위치할 수 있다.

- 데이터베이스 리포지토리 구현
- HTTP 클라이언트 구현
- 메시지 큐 어댑터
- 파일 저장소 구현
- 환경 변수 기반 설정 로더
- 시간, UUID, 랜덤 생성기 구현

클린 아키텍처에서 중요한 점은, 인프라가 애플리케이션을 참조하는 것은 가능하지만 그 반대는 피해야 한다는 것이다.
애플리케이션은 계약만 알고, 인프라는 그 계약을 구현한다.

현재는 `health/in_memory_status_reader.rs` 가 들어 있으며, 이것은 애플리케이션 레이어의 `HealthStatusReader` 포트를 구현하는 간단한 메모리 기반 어댑터다.
초기 부트스트랩과 테스트 환경에서 빠르게 조립하기 좋은 기본 구현으로 볼 수 있다.

### `crates/xavi-harness/`

이 폴더는 개발과 검증을 위한 하네스 레이어다.
일반적인 단위 테스트보다 한 단계 바깥에서, 여러 레이어를 묶어 시나리오 단위로 검증하는 구조를 만든다.

이 폴더를 별도 crate 로 둔 이유는 분명하다.

- 테스트용 조립 코드를 운영 코드와 분리할 수 있다.
- 반복적인 테스트 세팅을 재사용할 수 있다.
- 하네스를 중심으로 인수 테스트나 통합 테스트를 구조화하기 쉽다.
- 나중에 시뮬레이터, 픽스처 빌더, 테스트 더블 묶음을 확장하기 쉽다.

현재 구성에서:

- `src/builder.rs` 와 `src/harness.rs` 가 테스트용 composition root 역할을 나눠 맡는다.
- `src/fixtures/`, `src/doubles/`, `src/scenarios/`, `src/assertions/` 가 하네스 지원 구조를 이룬다.
- `tests/health_check.rs` 는 실제 하네스를 사용한 시나리오 테스트를 둔다.

향후 프로젝트가 커지면 이 하네스 레이어는 단순 테스트 도우미를 넘어서, 개발용 샌드박스나 시뮬레이션 런너 역할까지 맡을 수 있다.
사용자가 말한 "강하게 하네스 설계를 먼저 한다"는 방향과도 잘 맞는 구조다.

## `docs/` 하위 폴더 설명

### `docs/human/`

이 폴더는 사람 개발자를 위한 문서 공간이다.
배경 설명, 설계 이유, 의사결정 맥락처럼 사람이 이해하고 판단해야 하는 정보는 이곳에 우선 배치하는 것이 좋다.

대표적으로 다음 문서를 두기 좋다.

- 전체 아키텍처 설명서
- ADR(Architecture Decision Record)
- 기능 설계 문서
- 운영 절차 문서
- 장애 대응 가이드

이 폴더의 문서는 "왜 이렇게 설계했는가"를 충분히 설명해야 한다.
사람은 단순 규칙보다 맥락을 바탕으로 판단하기 때문에, 압축된 체크리스트만으로는 장기 유지보수 품질이 떨어질 수 있다.

이와 별도로 `user-docs/` 는 사용자에게 현재 코드 상태를 설명하는 특수 문서 구역으로 둘 수 있다.
이 구역은 인간 독자를 위한 산출물이지만, 운영은 사용자 문서 전담 AI 세션이 맡는다.

### `docs/human/user-docs/`

이 폴더는 사용자가 지금 코드베이스가 어떻게 작성되어 있는지를 바로 읽을 수 있게 만드는 실시간 문서 공간이다.
여기에는 "언제", "어떤 파일에", "무슨 목적으로", "어떤 기능의 코드가", "어떤 방식과 로직으로" 작성되어 있는지를 추적하는 문서가 들어간다.

이 폴더의 특징은 다음과 같다.

- 문서 독자는 사용자다.
- 문서 작성 주체는 사용자 문서 전담 AI 다.
- 다른 AI 역할 문서 시스템과 섞이지 않는다.
- 코드 설명을 빠르게 축적하는 용도로 운영한다.

### `docs/agent/`

이 폴더는 AI 에이전트 문서 시스템의 루트다.
중요한 점은 이 폴더 자체가 하나의 단일 작업 문서 공간이 아니라, 역할별로 분리된 여러 시스템의 입구라는 점이다.

대표적으로 다음 내용을 두기 좋다.

- 역할 분리 규칙
- 접근 경계 규칙
- 역할별 인덱스
- 공통 라우팅 규칙

실제 작업 문서는 다시 아래 네 폴더로 분리한다.

- `codegen/`
- `review/`
- `planning/`
- `ephemeral/`

핵심 원칙은 역할 간 문서 침범 금지다.
코드 생성 AI 는 코드 생성 문서만, 리뷰 AI 는 리뷰 문서만, 기획 AI 는 기획 문서만, 임시 AI 는 자신에게 배정된 임시 세션 문서만 읽고 관리해야 한다.
에이전트는 긴 배경 설명보다, 충돌 없이 작업하기 위한 명시 규칙을 더 효율적으로 활용하기 때문이다.

## 루트 주요 파일 설명

### `Cargo.toml`

루트 `Cargo.toml` 은 워크스페이스의 중심 설정 파일이다.
어떤 crate 들이 워크스페이스 멤버인지, 공통 Rust edition 이 무엇인지, 공통 lint 정책을 어떻게 가져갈지 같은 기준이 여기에 모인다.

이 파일은 단순 설정 파일이 아니라, 구조적 의도를 담는 선언문에 가깝다.
특히 멤버 목록과 lint 정책은 프로젝트가 어떤 품질 기준과 구조 경계를 중요하게 여기는지를 드러낸다.

### `README.md`

루트 `README.md` 는 프로젝트의 가장 짧은 입구 문서다.
처음 저장소를 열었을 때 무엇을 만들고 있는지, 어떤 레이어가 있는지, 어떤 명령부터 실행하면 되는지 빠르게 보여주는 역할을 한다.

세부 설계는 `docs/` 로 보내더라도, 루트 `README.md` 는 전체 방향을 간결하게 잡아주는 것이 좋다.

### `starter.md`

루트 `starter.md` 는 AI 에이전트의 공통 시작 문서다.
모든 AI 세션이 가장 먼저 읽는 라우팅 문서이며, 프로젝트 구조와 개발 방식, 그리고 자신이 접근해야 할 역할 문서 시스템을 안내한다.
즉 저장소 전체 관점에서 보면 `README.md` 는 사람용 짧은 입구이고, `starter.md` 는 AI 용 짧은 입구라고 이해하면 된다.

### `.gitignore`

버전 관리에서 제외할 파일을 정의한다.
빌드 산출물, 로컬 환경 파일, 운영체제 잡파일 등을 제외해 저장소를 깨끗하게 유지한다.

### `rustfmt.toml`

Rust 코드 포맷 규칙을 프로젝트 차원에서 통일하기 위한 파일이다.
팀 작업과 자동 생성 코드 품질을 일정하게 유지하는 데 중요하다.

## 의존 방향 정리

현재 구조의 의도된 의존 방향은 아래와 같다.

```text
apps/xavi-bootstrap
        ↓
crates/xavi-infrastructure
        ↓
crates/xavi-application
        ↓
crates/xavi-domain
```

그리고 `crates/xavi-harness` 는 검증 목적상 바깥쪽에서 여러 레이어를 조립해 사용할 수 있다.

```text
crates/xavi-harness
        ↓
crates/xavi-infrastructure
        ↓
crates/xavi-application
        ↓
crates/xavi-domain
```

핵심은 안쪽 레이어일수록 더 안정적이고, 바깥 레이어일수록 더 자주 바뀐다는 점이다.
구조를 이렇게 나누면 변경이 잦은 기술 선택이 도메인 규칙을 흔들지 않게 만들 수 있다.

## 폴더를 사용할 때의 실무 기준

새 파일을 추가할 때는 먼저 "이 파일이 어떤 책임을 가지는가"를 기준으로 위치를 정해야 한다.
편의상 가까운 폴더에 넣는 습관은 시간이 지나면 구조를 빠르게 무너뜨린다.

다음 기준으로 판단하면 된다.

- 핵심 개념과 규칙이면 `xavi-domain`
- 유스케이스와 포트면 `xavi-application`
- 외부 시스템 연결 구현이면 `xavi-infrastructure`
- 실행 진입점과 조립이면 `apps/xavi-bootstrap`
- 테스트 조립과 시나리오 검증이면 `xavi-harness`
- 사람이 읽는 설계 문서면 `docs/human`
- 에이전트가 읽는 작업 규칙이면 `docs/agent` 아래 해당 역할 폴더

이 기준을 엄격하게 유지할수록, 프로젝트는 커져도 무너지지 않는다.
