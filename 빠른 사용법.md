# 빠른 사용법

이 부트스트랩의 핵심은 `1회 작업 사이클`입니다.

AI에게 큰 작업을 한 번에 맡기지 않습니다.
작업을 작은 단위로 끊고, 각 단위마다 계획, 구현, 검수, 테스트, 보고서를 확인합니다.

목표는 단순합니다.

- AI가 무엇을 이해했는지 확인한다.
- 어떤 역할이 무엇을 했는지 확인한다.
- 어떤 코드가 왜 바뀌었는지 확인한다.
- 테스트와 검수가 실제로 있었는지 확인한다.
- 확인한 내용을 기준으로 다음 작업을 지시한다.

## 1회 작업 사이클이란?

한 사이클은 아래 흐름을 끝까지 마치는 단위입니다.

```text
사용자 요청
-> AI가 이해하고 작업 분해
-> 계획 작성
-> 사용자 승인
-> 구현
-> 검수
-> 테스트
-> 문서 정리
-> HTML 보고서 생성
-> 사용자가 보고서 확인
-> 다음 사이클 지시
```

즉, 코드를 만들었다고 바로 끝나는 것이 아닙니다.
HTML 보고서를 보고 무엇을 이해했고, 무엇을 바꿨고, 검증이 됐는지 확인한 뒤 다음 작업으로 넘어갑니다.

## 왜 작업 사이클로 개발하나?

- AI가 너무 많은 일을 한 번에 하다가 방향을 잃는 것을 줄입니다.
- 계획 없이 바로 구현하는 흐름을 막습니다.
- 검수와 테스트 없이 완료되는 일을 줄입니다.
- 어떤 요청 때문에 코드가 바뀌었는지 나중에 확인할 수 있습니다.
- 사용자가 확인한 뒤 다음 사이클로 넘어갈 수 있습니다.

## 실제 활용 방식

이 저장소는 아래 리듬으로 쓰는 것이 핵심입니다.

1. 사용자가 작은 목표 하나를 말합니다.
2. AI가 먼저 계획을 냅니다.
3. 사용자가 계획을 승인하거나 수정합니다.
4. AI가 구현, 검수, 테스트, 문서 정리를 진행합니다.
5. 마지막에 HTML 보고서가 열립니다.
6. 사용자는 보고서를 보고 다음 지시를 냅니다.

좋은 사이클은 작고 확인 가능해야 합니다.

- 좋은 예: "이번 사이클에서는 로그인 화면 UI만 만들어줘."
- 좋은 예: "이번 사이클에서는 문서 업로드 API만 구현해줘."
- 안 좋은 예: "전체 앱을 다 만들어줘."
- 안 좋은 예: "백엔드, 프론트, 배포까지 한 번에 끝내줘."

보고서를 볼 때는 아래 세 가지를 먼저 확인합니다.

1. AI가 내 요청을 제대로 이해했는가?
2. 실제 변경이 내가 승인한 계획과 맞는가?
3. 테스트와 검수 결과에 남은 문제가 있는가?

다음 지시는 보고서를 기준으로 냅니다.

```text
보고서에서 업로드 API는 괜찮아 보여.
다음 사이클에서는 업로드한 파일 목록을 보여주는 화면만 만들어줘.
계획 먼저 보고하고, 내가 승인하면 진행해줘.
```

## 처음 시작

AI coding tool에서 먼저 이렇게 요청합니다.

```text
starter.md 읽고 문서 체계 숙지하고, 프로젝트 구조를 간단히 파악해줘.
```

그 다음 만들고 싶은 프로젝트를 설명합니다.

```text
이 부트스트랩으로 <만들고 싶은 프로젝트>를 시작하고 싶어.

사용자:
<누가 쓰는지>

처음 필요한 기능:
<가장 먼저 만들 기능 2~3개>

starter.md 기준으로 필요한 절차대로 진행해줘.
```

## 개발 요청 예시

개발은 작게 요청하는 것이 좋습니다.

```text
이번 사이클에서는 <작은 기능 하나>만 구현해줘.
계획 먼저 보고하고, 내가 승인하면 진행해줘.
```

## 보고서 확인

각 사이클이 끝나면 HTML 보고서가 생성됩니다.
보고서에서는 아래 내용을 확인합니다.

- 사용자 요청 원문
- AI가 이해한 내용
- 역할별 지시와 반환
- 변경된 코드
- 테스트 결과
- 누락된 증거나 경고

이미 생성된 보고서는 아래 명령으로 다시 열 수 있습니다.

```bash
cargo run -p xavi-dev-console -- open-cycle --cycle-id <cycle-id>
```

## 주의

- `.xavi/` 안에는 로컬 작업 증거와 보고서가 남습니다.
- 비밀 키, 토큰, 비밀번호 같은 민감한 정보는 프롬프트에 넣지 않는 것이 좋습니다.
- 이 저장소는 완성 앱이 아니라, AI와 개발을 작은 작업 사이클로 관리하기 위한 부트스트랩입니다.
